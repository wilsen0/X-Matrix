这次不是 starter 壳子了，而是往前推进到了一个可跑 dry-run 流程的版本，新增了 4 个关键能力：
	1.	doctor
真的检查本地环境，不再只是打印提示。会探测：

	•	okx CLI 是否在 PATH
	•	~/.okx/config.toml 是否存在
	•	demo/live profile 是否大概率已配置
	•	当前发现了多少个 skills

	2.	plan
支持明确运行平面：

	•	research
	•	demo
	•	live

并且会把结果写进 runs/*.json，包含：
	•	skill chain
	•	proposals
	•	capability snapshot
	•	风险状态

	3.	apply
现在已经有真正的执行闸门了：

	•	research 下禁止写
	•	live 下必须 --approve
	•	默认是 dry-run
	•	只有加 --execute 才会尝试调用真实 okx CLI

	4.	policy-gate + official-executor
写操作已经收口成一条明确路径：

	•	自定义 skill 不直接写
	•	proposal 先过 policy
	•	过了才进入 official executor
	•	每次 apply 都会写回 run trace

我还把 skill frontmatter 也升级了，加入了：
	•	writes
	•	risk_level
	•	triggers

这让 runtime 对“模块化、可插拔、即时更新”的支持更像真的系统，而不是静态 demo。

我在容器里实际跑过这些命令：

npm run build
npm run doctor
npm run plan
node --loader ts-node/esm src/index.ts apply run_20260319163134 --proposal protective-put --approve --json
node --loader ts-node/esm src/index.ts apply run_20260319163203 --proposal protective-put --json

验证结果是：
	•	build 成功
	•	doctor 正常探测到当前容器没有本地 okx CLI，也没有 ~/.okx/config.toml
	•	demo plan 能生成 3 套 hedge proposals
	•	demo apply + --approve 会走完整 dry-run
	•	live apply 不带 --approve 会被 policy gate 拦住

这点我得说清楚：
真实下单路径我没有伪造成功。 因为当前容器里并没有你的本地 OKX CLI 和账户配置，所以我只能把它推进到“真实可执行框架 + 已验证 dry-run + 已验证 policy gate”的程度。这个边界是实的，不是我装出来的“可跑”。

你现在拿这个版本去讲，已经可以讲成：
	•	okx CLI 是唯一执行内核
	•	skill 是唯一扩展单位
	•	plan / apply / replay 都有审计痕迹
	•	live 写操作必须过 policy gate
	•	runtime 能根据 skill manifest 和本地能力快照动态工作

下一刀最值钱的，不是再加更多 skill，而是把示例 intents 替换成真实 OKX CLI 命令映射 + 实际 JSON 解析，把这个版本从“dry-run runtime”推进到“本地可接真环境的 demo-ready runtime”。
